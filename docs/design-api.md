## Approaches Considered

### Approach A: SQLite + Raw SQL
Fastest possible. Minimal scaffolding.
Pro: 2-day ship
Con: Raw SQL is error-prone, no type safety

### Approach B: PostgreSQL + Docker
"Production-ready" signal.
Pro: Shows you know real databases
Con: Setup complexity, more bugs, less time for polish

### Approach C: Prisma Hybrid (Chosen)
SQLite for dev speed, Prisma ORM for code quality, PostgreSQL optional in docker-compose.

## Recommended Approach

**Tech Stack:**
- Express (existing, lightweight)
- Prisma ORM (database-agnostic, type-safe, migrations)
- SQLite (dev), PostgreSQL (production-ready in docker-compose)
- Claude Anthropic SDK (real API calls)

**Database Schema (Prisma):**

```prisma
model Brief {
  id        String   @id @default(cuid())
  title     String
  description String
  channels  String[] // ["email", "linkedin", "medium"]
  status    String   @default("draft")
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
  drafts    Draft[]
  campaigns Campaign[]
}

model Draft {
  id        String   @id @default(cuid())
  briefId   String
  brief     Brief    @relation(fields: [briefId], references: [id], onDelete: Cascade)
  channel   String   // "email", "linkedin", "medium"
  content   String   // Generated draft text
  status    String   @default("pending_review")
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}

model Campaign {
  id        String   @id @default(cuid())
  briefId   String
  brief     Brief    @relation(fields: [briefId], references: [id], onDelete: Cascade)
  status    String   @default("draft")
  publishedAt DateTime?
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}
```

**API Endpoints (replace mock with real):**

1. `POST /briefs` — Create brief, call Claude API to generate drafts, return drafts
   - Input: title, description, channels (array)
   - Call Claude Anthropic SDK to generate channel-specific content
   - Store brief + drafts in database
   - Return: brief with generated drafts

2. `GET /briefs` — List all briefs (paginated)
   - Input: limit, offset
   - Return: briefs with draft count

3. `GET /briefs/:id` — Get brief with all drafts
   - Input: briefId
   - Return: brief + all related drafts

4. `POST /drafts/:id/approve` — Approve draft for publishing
   - Input: draftId
   - Update status to "approved"
   - Return: updated draft

5. `POST /campaigns` — Create campaign from approved drafts
   - Input: briefId, draftIds
   - Create campaign record
   - Return: campaign ID

6. `GET /campaigns/:id/analytics` — Return mock analytics (for now)
   - Input: campaignId, days
   - Return: email opens/clicks, LinkedIn engagement, Medium views (fake but realistic)

**Error Handling:**

- Claude API fails → return error with message "Draft generation failed, try again"
- Database down → return 500 "Database unavailable"
- Bad input → return 400 with field validation errors
- No channels specified → return 400 "At least one channel required"

## Open Questions

1. Should Claude API calls happen synchronously in POST /briefs, or async with status polling?

3. What happens if user edits a draft in Claude Desktop?
